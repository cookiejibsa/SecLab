Part 1 · Chapter 01

컴퓨터 기본 구조

어셈블리어는 CPU가 직접 이해하는 언어입니다.
그러므로 CPU가 어떻게 생겼는지, 메모리와 어떻게 대화하는지 먼저 알아야 합니다.

컴퓨터는 무엇으로 이루어져 있나?

컴퓨터는 크게 CPU(중앙처리장치), 메모리(RAM), 저장장치(SSD/HDD), 입출력장치로 이루어집니다.

CPU (Central Processing Unit, 중앙처리장치)

컴퓨터의 "뇌"입니다. 덧셈, 비교, 점프 같은 연산을 초당 수십억 번 실행합니다. 어셈블리 명령어는 결국 CPU에게 보내는 직접 지시입니다. "이 두 숫자를 더해라", "여기 주소의 값을 읽어라" 같은 명령들입니다.

RAM (Random Access Memory)

프로그램이 실행 중일 때 데이터를 임시로 저장하는 공간입니다. 책상 위의 메모지 같은 개념으로, 전원이 꺼지면 사라집니다. 어셈블리에서 mov로 읽고 쓰는 대부분의 공간이 이 RAM입니다. 주소(번지)로 접근하며, 모든 바이트는 고유한 주소를 가집니다.

저장장치 (SSD / HDD)

전원이 꺼져도 남아있는 영구 저장공간입니다. 파일들이 여기 있습니다. 프로그램을 실행하면 저장장치에서 RAM으로 복사되어 올라갑니다. SSD는 HDD보다 훨씬 빠르지만, 둘 다 RAM보다는 수십~수천 배 느립니다.

폰 노이만 구조 — 현대 컴퓨터의 설계 철학

폰 노이만 구조(Von Neumann Architecture)는 1945년에 만들어진 컴퓨터 설계 방식으로, 현재 여러분의 PC가 그대로 따르고 있습니다. 핵심 아이디어는 딱 하나입니다:

💡 폰 노이만의 핵심 아이디어

프로그램 코드(명령어)와 데이터를 같은 메모리에 저장한다. 덕분에 프로그램을 메모리에서 읽어와서 순서대로 실행할 수 있습니다. 어셈블리 코드도 결국 메모리의 특정 주소에 저장된 바이트들입니다.

비트(Bit)와 바이트(Byte)

비트 (Bit)

컴퓨터가 다룰 수 있는 가장 작은 정보 단위입니다. 0 또는 1, 두 가지 값만 가집니다. 왜 0과 1밖에 없냐면? 전기가 흐르면 1, 안 흐르면 0으로 물리적으로 표현하기 가장 쉽기 때문입니다.

바이트 (Byte)

8개의 비트를 묶은 단위입니다. 00000000부터 11111111까지, 즉 0부터 255까지의 값을 표현할 수 있습니다. 메모리의 주소는 바이트 단위로 매겨집니다. 1KB = 1024바이트, 1MB = 1024KB입니다.

수 체계와 진법

어셈블리 코드에는 숫자가 넘쳐납니다.
그런데 0x7fff, 0b1010 같은 이상한 표기가 나옵니다. 이게 뭔지 알아야 합니다.

우리가 쓰는 10진법(Decimal)

우리는 일상에서 10진법(Decimal)을 씁니다. 0~9까지 10개의 숫자를 쓰며, 9 다음에는 자리가 올라가 10이 됩니다. 예: 42, 255, 1000.

컴퓨터가 쓰는 2진법(Binary)

컴퓨터는 2진법(Binary)을 씁니다. 0과 1만 사용합니다. 1 다음에 바로 자리가 올라가서 10(이진수)이 됩니다.

어셈블리에서 자주 쓰는 16진법(Hexadecimal, Hex)

16진법(Hexadecimal)은 0~9, A~F까지 16개의 기호를 씁니다. A=10, B=11, C=12, D=13, E=14, F=15입니다. 왜 쓰냐고요? 2진수 4비트가 정확히 16진수 1자리에 대응되기 때문입니다. 8비트(1바이트)는 16진수 두 자리 FF로 표현됩니다. 훨씬 읽기 편합니다.

📌 표기법

C/어셈블리에서 0x를 앞에 붙이면 16진수입니다. 0xFF = 255, 0x41 = 65 = 문자 'A'. 메모리 주소는 대부분 0x7fffffffe5a0 같은 16진수로 표시됩니다.

음수 표현 — 2의 보수(Two's Complement)

컴퓨터는 음수를 어떻게 저장할까요? 2의 보수(Two's Complement) 방식을 사용합니다.

규칙은 간단합니다: 어떤 양수의 2의 보수를 구하려면
 ① 비트를 모두 뒤집고
 ② 1을 더합니다.

+5 = 0000 0101 (8비트 이진수)
 1단계: 비트 반전     → 1111 1010
 2단계: 1 더하기      → 1111 1011

따라서 -5 = 1111 1011 = 0xFB

왜 이 방식을 쓸까요? 덧셈 회로 하나로 덧셈과 뺄셈을 모두 처리할 수 있기 때문입니다. 5 + (-3)을 그냥 비트 덧셈으로 하면 정확히 2가 나옵니다. 컴퓨터 입장에서 매우 효율적입니다.

⚠️ 부호있는(Signed) vs 부호없는(Unsigned)

0xFF는 부호 없이 보면 255이고, 부호 있게 보면 -1입니다! 같은 비트 패턴이지만 해석이 달라집니다. 어셈블리에서 movsx(부호 확장)와 movzx(0 확장)의 차이가 여기서 생깁니다.

메모리 구조

프로그램이 실행될 때 메모리는 어떻게 나뉘어 있을까요? 이것을 모르면 스택 오버플로우, 버퍼 오버플로우 같은 개념을 이해할 수 없습니다.

메모리 주소란?

메모리 주소 (Memory Address)

RAM의 각 바이트는 고유한 번호(주소)를 가집니다. 아파트 호수처럼요. x86-64에서 주소는 64비트(8바이트)입니다. 보통 0x7fffffffe5a0 같은 16진수로 표시됩니다. 포인터(pointer)는 이 주소를 담는 변수입니다. C에서 int *p라고 쓸 때 p가 바로 주소를 담는 포인터입니다.

텍스트(Text) 세그먼트 — 코드가 사는 곳

여러분이 작성한 C/어셈블리 코드가 컴파일되면, 기계어 바이트들이 이 구역에 저장됩니다. 읽기 전용입니다. Ghidra나 IDA로 바이너리를 열면 이 구역을 분석합니다. rip 레지스터(명령 포인터)가 현재 실행 중인 명령의 주소를 가리킵니다.

데이터(Data) / BSS 세그먼트 — 전역변수

전역변수가 저장되는 공간입니다. 초기값이 있는 것은 Data, 없는 것은 BSS에 들어갑니다.

// C 코드
int initialized = 42;   // Data 세그먼트 (초기값 있음)
int uninitialized;       // BSS 세그먼트 (0으로 초기화됨)
📌 BSS란?

BSS는 "Block Started by Symbol"의 약자입니다(역사적 이유로 이상한 이름이 됐습니다). 실행 파일에는 실제 0 데이터를 넣지 않고 "이 크기만큼 0으로 채워진 공간이 필요하다"는 정보만 넣습니다. 덕분에 실행 파일 용량이 줄어듭니다.

힙(Heap) — 동적으로 할당하는 메모리

C의 malloc()으로 요청하는 메모리가 여기서 옵니다. 힙은 낮은 주소에서 높은 주소 방향으로 자랍니다. 직접 해제(free())하지 않으면 메모리 누수(Memory Leak)가 발생합니다. 리버싱에서 힙 취약점(Heap Exploitation)이 중요한 이유입니다.

스택(Stack) — 함수가 쓰는 임시 공간

가장 중요합니다. 함수를 호출할 때마다 스택에 새 공간이 생기고, 함수가 끝나면 사라집니다. 높은 주소에서 낮은 주소 방향으로 자랍니다. 즉, push하면 rsp(스택 포인터)가 감소합니다. 이 역방향 성장이 버퍼 오버플로우 공격의 핵심 요소입니다.

✅ 핵심 암기

스택은 위에서 아래로 자랍니다 (주소 감소). 힙은 아래에서 위로 자랍니다 (주소 증가). push rax는 rsp를 8 감소시키고 rax 값을 씁니다.

엔디안(Endianness) — 바이트 저장 순서

리틀 엔디안 (Little Endian)

x86/x86-64가 사용하는 방식입니다. 숫자를 메모리에 저장할 때 작은(낮은) 바이트가 낮은 주소에 먼저 옵니다. 예를 들어 0x12345678이라는 4바이트 값은 메모리에 78 56 34 12 순서로 저장됩니다. "리틀"은 "작은 쪽(낮은 바이트)이 먼저"라는 의미입니다.

값: 0x12345678 (32비트 정수)

CPU 작동 원리

어셈블리 명령어 하나가 CPU에서 어떻게 처리되는지 이해하면, 왜 레지스터가 필요한지, 왜 캐시가 중요한지 자연스럽게 이해됩니다.

명령어 사이클 — CPU가 하는 일의 반복

CPU는 매우 단순한 일을 엄청 빠르게 반복합니다. 이 반복을 명령어 사이클(Instruction Cycle)이라고 합니다:

  1. 패치(Fetch) — 명령어 가져오기
    rip(명령 포인터 레지스터)가 가리키는 메모리 주소에서 다음 명령어(기계어 바이트)를 읽어옵니다. 예: 주소 0x401234에서 48 89 c7 같은 바이트를 읽습니다.
  2. 디코드(Decode) — 명령어 해석
    읽어온 바이트가 무슨 명령인지 해석합니다. 48 89 c7이 mov rdi, rax라는 명령임을 알아냅니다. 명령어마다 길이가 다릅니다(1~15바이트). 이 해석 과정을 디코딩이라고 합니다.
  3. 실행(Execute) — 명령 수행
    ALU(산술논리장치)가 실제 연산을 수행합니다. add rax, rbx라면 두 레지스터 값을 더하는 회로가 작동합니다.
  4. 라이트백(Writeback) — 결과 저장
    연산 결과를 레지스터나 메모리에 씁니다. 플래그 레지스터(rflags)도 이때 갱신됩니다.

캐시(Cache) — CPU와 RAM 사이의 고속 메모리

캐시 (Cache)

RAM은 CPU에 비해 매우 느립니다. 그래서 CPU 내부에 더 빠른 소용량 메모리(캐시)를 두고, 자주 쓰는 데이터를 미리 복사해둡니다. L1(가장 빠름, 작음) → L2 → L3 순으로 계층이 있습니다. 어셈블리 최적화에서 "캐시 히트/미스"가 중요한 성능 요소입니다.

📌 사이클이란?

사이클(Cycle)은 CPU 클럭 한 번의 단위입니다. 3GHz CPU는 1초에 30억 번 클럭이 뜁니다. 1 사이클 = 약 0.33나노초(0.00000000033초)입니다. 레지스터는 1사이클이면 끝나지만, RAM은 200사이클이나 기다려야 합니다. 그래서 레지스터 접근이 압도적으로 빠릅니다.

Part 1 · Chapter 05

운영체제 기초

어셈블리로 시스템 콜을 하거나, 리버싱으로 프로그램을 분석할 때 운영체제가 하는 역할을 알아야 합니다.

커널(Kernel)과 유저 공간(User Space)

커널 (Kernel)

운영체제의 핵심입니다. 하드웨어(메모리, 파일시스템, 네트워크)를 직접 관리하며, 특권 명령어를 실행할 수 있습니다. "링 0(Ring 0)"이라는 최고 권한 모드에서 실행됩니다.

유저 공간 (User Space)

우리가 만드는 프로그램이 실행되는 공간입니다. 직접 하드웨어를 건드릴 수 없고, 커널에 요청(시스템 콜)을 해야 합니다. "링 3(Ring 3)"에서 실행됩니다. 이 구분이 보안의 핵심입니다.

시스템 콜(System Call)

유저 프로그램이 커널에 서비스를 요청하는 방법입니다. C의 printf()는 내부적으로 write 시스템 콜을 씁니다. 어셈블리에서는 syscall 명령어로 직접 호출합니다. 자세한 사용법은 Chapter 15에서 다룹니다.

📌 왜 시스템 콜이 필요한가?

만약 모든 프로그램이 하드웨어를 마음대로 건드릴 수 있다면? 악성 프로그램이 다른 프로그램의 메모리를 훔쳐보거나, 디스크를 마음대로 지울 수 있습니다. 커널이 문지기 역할을 해서 "파일을 읽고 싶다면 나(커널)한테 요청해라"라고 통제합니다.

프로세스와 쓰레드

프로세스 (Process)

실행 중인 프로그램의 인스턴스입니다. 자체 메모리 공간(주소 공간)을 가집니다. 다른 프로세스의 메모리를 함부로 읽을 수 없습니다 (OS가 가상 메모리로 분리).

쓰레드 (Thread)

프로세스 안에서 동시에 실행되는 여러 실행 흐름 중 하나입니다. 같은 프로세스 내 쓰레드들은 메모리를 공유합니다. 예: 웹 브라우저에서 동영상 재생 쓰레드, 사용자 입력 처리 쓰레드가 동시에 돌아갑니다. 어셈블리에서는 각 쓰레드마다 고유한 스택이 있습니다.

가상 메모리 (Virtual Memory)

각 프로세스는 메모리 전체를 혼자 쓰는 것처럼 보이는 가상 주소 공간을 받습니다. 실제 RAM 주소와의 변환은 MMU(메모리 관리 장치)가 담당합니다. 그래서 다른 프로그램과 주소가 충돌하지 않습니다.

✅ 리버싱에서 중요한 이유

리버싱 도구(gdb, x64dbg)로 프로세스를 분석할 때 보이는 주소들은 가상 주소입니다. ASLR(주소 공간 배치 난수화)이 켜져 있으면 실행할 때마다 주소가 달라집니다. 이것도 보안 기법 중 하나입니다.

어셈블리 파일의 전체 구조

C에서 #include와 main()이 있듯, 어셈블리 파일도 반드시 지켜야 하는 기본 뼈대가 있습니다.
이 챕터에서는 "빈 파일에서 실행 가능한 프로그램까지"의 전체 틀을 C와 1:1로 비교해서 설명합니다.

C 파일 구조 vs 어셈블리 파일 구조 — 한눈에 비교

C 파일 전체 구조
// ① 헤더 포함 (라이브러리 가져오기)
#include <stdio.h>

// ② 전역 변수 선언
int global_var = 42;

// ③ 함수 정의
void say_hello() {
    printf("Hello\n");
}

// ④ 진입점 (프로그램 시작)
int main() {
    say_hello();
    return 0;  // ⑤ 종료
}
어셈블리 파일 전체 구조 (NASM)
; ① section .data — 전역 데이터 / 문자열
section .data
    msg db "Hello", 0x0a
    msg_len equ $ - msg

; ② section .bss — 초기화 안 된 버퍼
section .bss
    buf resb 64

; ③ section .text — 실행 코드
section .text
    global _start  ; 진입점 외부 공개

; ④ 함수 정의
say_hello:
    mov  rax, 1
    mov  rdi, 1
    mov  rsi, msg
    mov  rdx, msg_len
    syscall
    ret

; ⑤ 진입점 (프로그램 시작)
_start:
    call say_hello
    mov  rax, 60  ; ⑥ exit(0)
    xor  rdi, rdi
    syscall

① 섹션(Section) — C의 영역 구분 대신 파일을 나눈다

섹션 (Section)

어셈블리 파일은 용도에 따라 공간을 나눕니다. 이 구역 하나하나를 섹션(Section)이라고 합니다. C에서 #include는 "이 라이브러리를 가져와라"는 선언이라면, 섹션 선언은 "여기서부터는 코드다 / 여기서부터는 데이터다"를 구분하는 선언입니다. 어셈블러가 이 구분을 읽고 실행 파일을 올바르게 만들어냅니다.

섹션C에 대응하는 것내용특성
section .text함수 본문, 실행 코드CPU가 실행하는 명령어들읽기 전용, 실행 가능
section .dataint x = 5; 초기화 전역변수초기값이 있는 데이터, 문자열읽기/쓰기 가능
section .bssint buf[64]; 미초기화 전역0으로 채워지는 공간 예약실행 파일 크기에 안 잡힘
📌 .bss와 .data의 차이가 왜 중요한가요?

.data에 쓴 값은 실행 파일 안에 실제 바이트로 저장됩니다. 반면 .bss는 "이 크기만큼 빈 공간이 필요하다"는 정보만 저장하고, 실제 0 바이트는 프로그램이 시작할 때 OS가 채워줍니다. 덕분에 char buf[1024*1024];처럼 큰 버퍼를 선언해도 실행 파일 크기가 늘지 않습니다.

② .data 섹션 — 데이터 선언 문법

C에서 int x = 42;처럼 변수를 선언하듯, .data 섹션에는 데이터를 선언합니다.

C 전역변수 선언
char  msg[] = "Hello\n";
int   count = 10;
long  big   = 0x1234;
short small  = 5;
NASM .data 선언
section .data
    msg   db "Hello", 0x0a, 0  ; byte(s)
    count dd 10              ; 4바이트 정수
    big   dq 0x1234          ; 8바이트 정수
    small dw 5               ; 2바이트 정수
지시어크기의미C 타입예시

dd4바이트Define Dwordintnum dd 42
dq8바이트Define Qwordlongptr dq 0
📌 equ와 $ — 상수와 현재 위치

equ는 C의 #define과 같은 상수 정의입니다. $는 "현재 어셈블리 위치"를 뜻합니다. len equ $ - msg는 현재 위치에서 msg 시작 위치를 빼서 msg의 바이트 수를 자동으로 계산하는 관용구입니다.

③ .bss 섹션 — 버퍼/배열 예약

C 미초기화 전역변수
char  buf[64];
int   arr[10];
long  bigbuf[8];
NASM .bss 선언
section .bss
    buf    resb 64   ; 64 × 1바이트
    arr    resd 10   ; 10 × 4바이트
    bigbuf resq 8    ; 8 × 8바이트
지시어크기의미C 대응
resb NN × 1바이트Reserve Bytechar buf[N]
resw NN × 2바이트Reserve Wordshort arr[N]
resd NN × 4바이트Reserve Dwordint arr[N]
resq NN × 8바이트Reserve Qwordlong arr[N]

④ 진입점 선언 — global _start

_start와 global — 프로그램의 출발점

C의 int main()처럼, 어셈블리에서는 _start 레이블이 "프로그램이 여기서 시작한다"는 뜻입니다. global _start는 이 레이블을 링커에게 공개하는 선언입니다. ld(링커)가 이 이름을 찾아서 프로그램의 시작 주소로 지정합니다.

📌 _start vs main — 언제 무엇을 쓰나?

순수 어셈블리로 직접 링크할 때 (ld 사용): _start를 씁니다.
C 표준 라이브러리(libc)와 함께 링크할 때 (gcc 사용): main을 씁니다.

gcc를 쓰면 libc가 자동으로 _start를 만들고 그 안에서 초기화 후 main을 호출해줍니다. 리버싱할 때 _start → __libc_start_main → main 순서로 호출되는 이유가 바로 이 때문입니다.

⑤ 함수 정의 — 레이블 + 본문 + ret

C에서 void func() { ... }처럼 함수를 만들 수 있듯, 어셈블리에서도 함수를 정의합니다. 레이블(이름) + 프롤로그 + 본문 + ret(끝)이 기본 구조입니다.

C 함수 정의
// 반환값 없는 함수
void greet() {
    printf("Hi\n");
}  // 닫는 } = 함수 끝

// 값 반환 함수
int add(int a, int b) {
    return a + b;
}
어셈블리 함수 정의
; void greet()
greet:               ; ← 함수 이름 = 레이블
    push rbp
    mov  rbp, rsp   ; 프롤로그
    ; ... 함수 내용 ...
    leave
    ret              ; ← 함수 끝 = ret

; int add(int a, int b)
; a=rdi, b=rsi, 반환값=rax
add:
    lea  eax, [edi+esi]
    ret
📌 함수 정의 핵심 3가지

① 레이블: 함수 이름 + 콜론(greet:). 이 주소가 함수의 시작점입니다.
② 프롤로그/에필로그: push rbp; mov rbp, rsp로 시작, leave로 끝. 짧은 함수는 생략 가능합니다.
③ ret: 함수 종료 명령. C의 닫는 중괄호 }에 해당합니다. 반드시 있어야 합니다.

⑥ 프로그램 종료 — C의 return 0에 대응

C에서 main()이 return 0으로 끝나듯, 어셈블리에서는 반드시 exit 시스템 콜로 명시적으로 끝내야 합니다. 없으면 CPU가 코드 다음에 있는 메모리를 계속 실행하다가 충돌합니다.

C 종료
int main() {
    // ... 코드 ...
    return 0;  // 정상 종료
}
어셈블리 종료 (필수!)
; exit(0) — Linux x86-64
mov  rax, 60      ; syscall 번호 60 = exit
xor  rdi, rdi    ; 종료 코드 0 (정상)
syscall          ; 커널에 종료 요청

; libc와 함께 쓸 때 (main에서 ret)
xor  eax, eax    ; return 0
ret

레지스터 완전 해부

레지스터는 CPU 안에 있는 초고속 저장 공간입니다.
어셈블리의 모든 연산은 레지스터를 중심으로 돌아갑니다.

레지스터란?

레지스터 (Register)

CPU 안에 있는 극소량의 저장 공간입니다. RAM보다 수백 배 빠릅니다. x86-64에는 16개의 범용 레지스터가 있으며, 각각 64비트(8바이트)입니다. C 변수를 메모리에 저장하는 것과 달리, 어셈블리에서는 가능한 한 레지스터를 이용해 연산합니다.

레지스터 크기별 이름 — 같은 물리 레지스터, 다른 이름

중요합니다! x86-64의 레지스터는 크기별로 다른 이름을 가집니다. rax의 하위 32비트가 eax, 하위 16비트가 ax, 하위 8비트가 al입니다. 이들은 같은 물리 공간의 일부입니다:

📌 이름의 의미

이름에는 역사가 있습니다. RAX = Register AX (64비트 확장), EAX = Extended AX (32비트), AX = 16비트 원래 이름, AH = AX High (상위 8비트), AL = AX Low (하위 8비트). e(Extended/확장), r(Register/64비트 시대)이 붙은 것입니다.

⚠️ 32비트 쓰면 상위가 0이 됩니다

mov eax, 1을 실행하면 rax의 상위 32비트는 자동으로 0이 됩니다. 이건 x86-64의 중요한 특성입니다. 반면 mov ax, 1은 상위 48비트를 건드리지 않습니다.

어셈블리 명령어는 수백 개가 있지만, 실제 코드의 80%는 여기 나오는 명령어들로 이루어집니다.
각 명령어를 C와 1:1로 대응시켜 이해합시다.

어셈블리 명령어 기본 형식

명령어  목적지, 소스
; 예:
mov  rax, rbx    ; rax = rbx  (rbx를 rax로 복사)
add  rax, 5      ; rax = rax + 5
sub  rdi, rsi    ; rdi = rdi - rsi
📌 Intel 문법 vs AT&T 문법

어셈블리에는 두 가지 문법이 있습니다. Intel 문법: mov rax, rbx (목적지 먼저), AT&T 문법: movq %rbx, %rax (소스 먼저, % 붙음). 이 교재는 Intel 문법을 씁니다. Ghidra도 기본적으로 Intel 문법입니다. GDB는 기본이 AT&T이지만 set disassembly-flavor intel로 바꿀 수 있습니다.

LEA — 주소 계산의 만능 도구

LEA (Load Effective Address)는 겉보기엔 주소를 계산하는 명령어이지만, 컴파일러가 곱셈/덧셈을 할 때도 씁니다!

플래그 레지스터와 조건 점프

어셈블리에서 if문과 반복문은 플래그와 조건 점프 명령어로 만들어집니다.
이것이 어셈블리의 제어 흐름(Control Flow)입니다.

RFLAGS — 상태 플래그 레지스터

RFLAGS는 연산 결과의 "상태"를 기억하는 특별한 레지스터입니다. 각 비트가 하나의 플래그를 나타냅니다. cmp나 add 같은 연산 후 자동으로 갱신됩니다.

ZF
Zero Flag
연산 결과가 0이면 1. cmp a, b에서 a==b이면 두 값의 차가 0이므로 ZF=1.
CF
Carry Flag
덧셈에서 올림수(carry) 발생 시, 또는 뺄셈에서 빌림(borrow) 발생 시 1. 부호없는 비교에 사용.
SF
Sign Flag
결과의 최상위 비트(부호 비트)와 같습니다. 음수면 SF=1.
OF
Overflow Flag
부호있는 연산에서 오버플로우(표현 범위 초과) 발생 시 1.
PF
Parity Flag
결과의 하위 8비트에서 1인 비트 수가 짝수면 1. 현대에는 잘 안 씁니다.
DF
Direction Flag
문자열 명령어 방향. 0=증가, 1=감소. cld/std로 설정.

스택(Stack) 동작 상세

스택은 함수 호출, 지역 변수, 리턴 주소를 관리하는 핵심 구조입니다.
스택을 완전히 이해하면 버퍼 오버플로우 공격도 이해됩니다.

스택이란? — LIFO 자료구조

LIFO (Last In, First Out)

마지막에 넣은 것이 먼저 나옵니다. 책상에 쌓은 책처럼요. push로 맨 위에 놓고, pop으로 맨 위에서 꺼냅니다. rsp(스택 포인터)는 항상 현재 스택의 "맨 위"(가장 낮은 주소)를 가리킵니다. LIFO는 자료구조의 일종입니다.

PUSH와 POP의 정확한 동작

PUSH rax 동작
// 등가 C 코드
rsp -= 8;        // 스택 포인터 감소
*(long*)rsp = rax; // 값 저장
POP rax 동작
// 등가 C 코드
rax = *(long*)rsp; // 값 읽기
rsp += 8;         // 스택 포인터 증가

스택은 높은 주소에서 낮은 주소로 자랍니다. push할수록 rsp는 작아집니다.

CALL과 RET — 함수 호출의 본질

CALL 명령어

call func는 두 가지 일을 합니다:
 ① 현재 rip(다음 명령어 주소)를 스택에 push
 ② func 주소로 점프
즉 push rip; jmp func와 같습니다. 스택에 저장된 이 주소를 리턴 주소(Return Address)라고 합니다. 함수가 끝난 후 돌아올 위치를 기억하는 것입니다.

RET 명령어

ret는 스택 최상단 값을 rip에 넣고 rsp를 8 증가시킵니다. 즉 pop rip와 같습니다. call이 저장해둔 리턴 주소로 돌아갑니다.

🔴 보안: 버퍼 오버플로우의 원리

스택에는 "로컬 변수"와 "리턴 주소"가 인접해 있습니다. 로컬 버퍼(예: 64바이트 배열)에 경계 검사 없이 데이터를 쓰면, 버퍼를 넘쳐서 리턴 주소를 덮어쓸 수 있습니다. ret 시 공격자가 원하는 주소로 점프됩니다. 이것이 고전적인 스택 기반 버퍼 오버플로우(Stack-based Buffer Overflow)입니다. 자세한 내용은 Chapter 20에서 다룹니다.

함수 호출 규약 (Calling Convention)

여러 함수가 통신하려면 약속이 필요합니다.
"인수를 어디에 넣어서 넘길까?", "리턴값은 어디에?", "어떤 레지스터를 보존해야 하나?" 이 약속이 호출 규약입니다.

📌 인수(Argument)와 매개변수(Parameter)의 차이

매개변수(Parameter): 함수를 정의할 때 쓰는 변수 이름입니다. int add(int a, int b)에서 a, b가 매개변수.

리턴값

정수/포인터 리턴값은 rax에, 128비트 리턴은 rax:rdx에 넣습니다. 부동소수점 리턴은 xmm0에 넣습니다.

Caller-saved vs Callee-saved 레지스터

Caller-saved (호출자가 저장)

함수를 호출하기 전에 내가(caller가) 저장해야 하는 레지스터입니다. 피호출 함수가 마음대로 수정할 수 있습니다. rax, rcx, rdx, rsi, rdi, r8, r9, r10, r11. Caller는 '부르는 쪽', Callee는 '불리는 쪽'입니다.

Callee-saved (피호출자가 저장)

호출된 함수가 사용하기 전에 저장하고, 리턴 전에 복원해야 하는 레지스터입니다. 호출자는 이 레지스터들이 함수 호출 전후로 변하지 않는다고 믿을 수 있습니다. rbx, rbp, r12, r13, r14, r15

C 코드
int add(int a, int b) {
    return a + b;
}

int main() {
    int r = add(3, 4);
}
어셈블리
add:                   ; int add(int a, int b)
    lea  eax, [edi+esi]  ; a+b → eax (리턴값)
    ret

main:
    mov  edi, 3          ; 1번째 인수 a=3
    mov  esi, 4          ; 2번째 인수 b=4
    call add             ; 함수 호출
    ; rax = 7 (리턴값)

함수 프롤로그(Prologue)와 에필로그(Epilogue)

모든 함수는 시작과 끝에 정해진 패턴을 따릅니다. 이것을 프롤로그(시작 의식)와 에필로그(끝 의식)라고 합니다. 리버싱에서 함수 경계를 찾을 때 이 패턴을 봅니다.

C 함수
long example(long x) {
    long a = x * 2;
    long b = a + 1;
    return b;
}
어셈블리 (rdi=x)
example:
    ; ── 프롤로그 ──
    push rbp         ; 이전 rbp 저장
    mov  rbp, rsp   ; rbp = 현재 스택 위치
    sub  rsp, 16    ; 로컬변수 공간 16바이트 확보
    ; ── 함수 본문 ──
    lea  rax, [rdi*2]
    mov  [rbp-8],  rax  ; a = x*2
    add  rax, 1
    mov  [rbp-16], rax  ; b = a+1
    mov  rax, [rbp-16]  ; 리턴값 준비
    ; ── 에필로그 ──
    leave            ; mov rsp,rbp; pop rbp
    ret              ; 리턴 주소로 점프
📌 leave 명령어

leave는 mov rsp, rbp + pop rbp를 합친 단축 명령어입니다. 에필로그에서 스택을 정리할 때 씁니다. enter 명령어도 있지만 거의 사용 안 합니다.

Frame Pointer Omission (FPO) — 최적화 시

컴파일러가 최적화(-O2)를 하면 rbp를 프레임 포인터로 쓰지 않습니다. 대신 rsp 기준으로 지역변수를 접근합니다. 프롤로그에서 push rbp; mov rbp, rsp가 없어집니다. 리버싱에서 프레임이 없는 함수를 만날 수 있으니 알아두세요.

컴파일러 최적화 패턴

실제 바이너리를 리버싱하면 사람이 직접 짠 어셈블리와 달리 이상하게 생긴 코드들이 나옵니다. 이것들은 컴파일러 최적화의 결과입니다.

나눗셈 → 곱셈+시프트 변환

나눗셈(idiv)은 매우 느린 명령어입니다. 컴파일러는 상수로 나누는 경우 곱셈과 시프트로 대체합니다.

C 코드 (원본 의도)
int x = n / 7;
컴파일러가 생성한 코드
; n / 7 최적화
mov   eax, edi         ; n
mov   ecx, 0x92492493  ; 마법의 상수(Magic Number)
imul  ecx              ; edx:eax = n * 상수
add   edx, edi
sar   edx, 2            ; 오른쪽 시프트
sar   edi, 31           ; 부호 처리
sub   edx, edi
; edx = n / 7
✅ 리버싱 팁

이런 "마법의 상수" 패턴을 보면 나눗셈입니다. 온라인 도구 "MagicDivider Calculator"나 Ghidra의 디컴파일러가 자동으로 원래 나눗셈으로 복원해줍니다.

모듈로(%) 최적화

C 코드

; n % 8 = n & 7  (2^3 - 1 = 0b0111)
; AND로 하위 3비트만 남기면 0~7 반복 = 나머지와 같음
and  eax, 7

; n % 16 = n & 15 (0xF)
and  eax, 0xF
📌 왜 AND가 나머지와 같나요?

2의 거듭제곱 N으로 나눈 나머지는 하위 log₂(N)비트와 같습니다. 예: 13 % 8 → 13 = 0b1101, 하위 3비트 = 0b101 = 5. 실제로 13 ÷ 8 = 1 나머지 5. and eax, 7이 하위 3비트를 남깁니다 (7 = 0b0111).

테일 콜 최적화 (Tail Call Optimization, TCO)

함수의 마지막 동작이 다른 함수 호출일 때, call + ret 대신 jmp로 대체합니다. 스택 프레임을 새로 쌓지 않아 효율적이고, 스택 오버플로우를 방지합니다.

테일 콜 (Tail Call)

테일 콜(Tail Call)은 함수의 마지막 동작(꼬리, tail)이 다른 함수를 호출하는 것입니다. 이 경우 현재 스택 프레임이 더 이상 필요 없으므로, call/ret 대신 jmp로 바꿔 스택을 재활용합니다. 재귀 함수 최적화에서 특히 중요합니다.

C 코드 (꼬리 재귀)
// 꼬리 재귀: 마지막이 자기 호출
void foo(int x) {
    if (x == 0) return;
    bar(x - 1); // 마지막 동작
}
TCO 적용 (call→jmp)
foo:
    test edi, edi   ; x == 0?
    jz   .done
    dec  edi        ; x - 1
    jmp  bar        ; call+ret 대신 jmp!
                    ; bar가 foo의 caller로 직접 ret
.done:
    ret

조건부 이동 — CMOV (Conditional Move)

분기 예측 실패(branch misprediction)를 피하기 위해 컴파일러는 jmp 대신 조건부 이동 명령어를 씁니다. cmov는 플래그 조건이 맞을 때만 복사를 수행하므로 분기 없이 조건부 대입이 가능합니다.

C 코드
// 삼항 연산자 → CMOV 최적화
int max = (a > b) ? a : b;
int abs_val = (x < 0) ? -x : x;
CMOV 최적화 (분기 없음)
; rdi=a, rsi=b → rax=max
cmp   rdi, rsi    ; a - b, 플래그 설정
mov   rax, rsi    ; rax = b (기본값)
cmovg rax, rdi   ; a>b이면 rax = a

; rdi=x → rax=abs(x)
mov   rax, rdi    ; rax = x
neg   rax         ; rax = -x
cmovns rax, rdi ; x>=0이면 rax = x (원래값)
명령어조건C 연산자 유사
cmove / cmovzZF=1 (같음)== 0
cmovne / cmovnzZF=0 (다름)!= 0
cmovg부호있음 >a > b
cmovl부호있음 <a < b
cmovsSF=1 (음수)x < 0
cmovnsSF=0 (양수/0)x >= 0

루프 언롤링 (Loop Unrolling)

반복문을 여러 번 펼쳐서 반복 횟수 자체를 줄이는 최적화 기법입니다. 루프 오버헤드(반복 카운터 증가, 조건 체크, 점프)를 줄입니다.

xmm0~xmm15128비트SSE ~ SSE4.24×float, 2×double, 16×byte 동시 처리
ymm0~ymm15256비트AVX, AVX28×float, 4×double, 32×byte 동시 처리
zmm0~zmm31512비트AVX-51216×float, 8×double, 64×byte 동시 처리
📌 XMM ↔ YMM ↔ ZMM 관계

xmm0는 ymm0의 하위 128비트이고, ymm0는 zmm0의 하위 256비트입니다. 즉 물리적으로 같은 레지스터의 일부를 다른 이름으로 부릅니다. VEX 인코딩을 쓰는 AVX 명령어로 ymm/xmm을 쓰면 상위 비트가 0으로 정리됩니다.

접미사의미예시 명령어설명
psPacked Single-precision floataddps, mulps, divpsfloat 4개 동시 처리

float a[4] = {1,2,3,4};

addps  xmm0, xmm1   ; xmm0 = [6|8|10|12]
                       ; 4개 덧셈이 1 사이클에!
movaps [rdx], xmm0   ; c에 저장
📌 movaps vs movups

movaps는 16바이트 정렬된 주소에서만 작동합니다. movups는 비정렬 주소에서도 됩니다. 리버싱에서 movups가 더 자주 보입니다.

SSE2 최적화 버전
; 16바이트를 한 번에 비교
movdqu xmm0, [rdi]     ; 문자열1에서 16바이트 로드
movdqu xmm1, [rsi]     ; 문자열2에서 16바이트 로드

C → ASM 패턴 사전

리버싱을 할 때 어셈블리를 보고 원래 C 코드를 추론하는 능력이 핵심입니다. 자주 나오는 패턴과 각 패턴을 인식하는 방법을 정리합니다.

switch-case → 점프 테이블

점프 테이블 (Jump Table)

여러 case의 코드 주소를 배열로 만들어놓고, 인덱스로 한 번에 해당 case로 점프하는 방식입니다. if-else 체인보다 훨씬 빠릅니다. Ghidra에서 jmp [rax*8 + 주소] 패턴이 보이면 switch문을 의심하세요.

C 코드
switch (x) {
  case 0: foo(); break;
  case 1: bar(); break;
  case 2: baz(); break;
  default: def(); break;
}
점프 테이블 방식
; rdi = x
cmp   rdi, 2             ; x > 2?
ja    .default           ; 범위 벗어나면 default
jmp   [jmp_table + rdi*8]; 테이블로 점프!

jmp_table:
    dq .case0            ; [0] = case0 주소
    dq .case1            ; [1] = case1 주소
    dq .case2            ; [2] = case2 주소

.case0: call foo; jmp .end
.case1: call bar; jmp .end
.case2: call baz; jmp .end
.default: call def
.end:

함수 포인터 → 간접 호출

C 코드
// 함수 포인터
void (*fp)(int) = get_handler();
fp(42);

// 함수 포인터 배열
void (*handlers[3])() = {a, b, c};
handlers[i]();
어셈블리 (간접 호출)
; 함수 포인터
call get_handler     ; rax = 함수 주소
mov  edi, 42
call rax             ; 간접 호출!

; 함수 포인터 배열
; rbx=handlers, rcx=i
call [rbx + rcx*8]  ; handlers[i]()
✅ 리버싱 팁: call reg 패턴

call rax, call [rbx+8] 같은 간접 호출은 함수 포인터 또는 C++ vtable입니다. GDB에서 동적 분석으로 call rax 직전에 rax 값을 확인하면 실제 호출 대상을 알 수 있습니다.

C++ 가상 함수 → vtable 패턴

vtable (Virtual Function Table, 가상 함수 테이블)

C++에서 virtual 함수가 있는 클래스는 vtable을 가집니다. vtable은 해당 클래스의 virtual 함수 주소들이 담긴 배열입니다. 각 객체는 자신의 vtable 주소(vptr)를 첫 8바이트에 저장합니다.

; obj->method() 호출 (obj = rdi = this 포인터)
mov  rax, [rdi]         ; vptr 로드 (객체 첫 8바이트 = vtable 주소)
call [rax]              ; vtable[0] = 첫 번째 virtual 함수
; 또는
call [rax + 0x10]      ; vtable[2] = 세 번째 virtual 함수 (0x10 = 2*8)

; 리버싱 팁: vtable 오프셋 → 몇 번째 virtual 함수인지 알 수 있음
; 0x00 = 첫 번째, 0x08 = 두 번째, 0x10 = 세 번째 ...

NULL 체크 패턴

C 코드
if (p != NULL) {
    *p = 10;
}
// NULL = 0, 아무것도 안 가리키는 포인터
어셈블리
test rdi, rdi    ; rdi & rdi → ZF=1이면 0(NULL)
jz   .skip         ; NULL이면 건너뜀
mov  QWORD PTR [rdi], 10
.skip:
📌 test reg, reg 패턴

test rax, rax는 cmp rax, 0과 동일한 효과지만 기계어가 1바이트 짧습니다. 컴파일러가 항상 이 방법을 선택합니다. NULL 역참조(Segfault)는 리버싱 대상에서 자주 보이는 취약점입니다.

구조체 반환 (여러 값 리턴)

C 코드
typedef struct { long a, b; } Pair;
Pair make_pair(long x) {
    return (Pair){x, x+1};
}
어셈블리 (rax+rdx 리턴)
make_pair:          ; rdi = x
    mov  rax, rdi   ; a = x → rax
    lea  rdx, [rdi+1] ; b = x+1 → rdx
    ret
; 2개의 long이 rax, rdx에 각각 리턴됨!

가변 길이 배열(VLA) / alloca 패턴

C 코드
void foo(int n) {
    // 런타임에 크기 결정
    int arr[n];
    // alloca로도 가능
}
어셈블리
foo:                 ; rdi = n
    push rbp
    mov  rbp, rsp
    ; n * sizeof(int) 바이트 동적 확보
    lea  rax, [rdi*4+15]  ; 정렬 패딩 추가
    and  rax, -16    ; 16바이트 정렬
    sub  rsp, rax    ; rsp 동적 감소
    ; arr = rsp

정적 분석 vs 동적 분석

정적 분석 (Static Analysis)

프로그램을 실행하지 않고 분석합니다. Ghidra, IDA Pro로 디스어셈블/디컴파일합니다. 전체 코드 흐름 파악에 좋지만, 난독화나 런타임에 생성되는 코드는 분석이 어렵습니다.

동적 분석 (Dynamic Analysis)

프로그램을 실행하면서 분석합니다. GDB, x64dbg로 브레이크포인트를 설정하고 레지스터/메모리를 관찰합니다. 실제 실행 흐름, 암호화 키, 동적 생성 코드를 볼 수 있습니다.

Ghidra 사용 기초

  1. 프로젝트 생성 및 바이너리 임포트
    File → New Project → Non-Shared → 이름 설정. 그 다음 File → Import File로 바이너리를 불러옵니다. ELF(Linux 실행파일), PE(Windows .exe), Mach-O(macOS) 모두 지원합니다.
  2. 자동 분석 실행
    임포트 후 "Analyze Now?" 클릭 → 기본 설정 OK. 수십 초 걸립니다. 함수 자동 인식, 심볼 복구, 문자열 추출이 됩니다.
  3. Symbol Tree에서 함수 탐색
    좌측 Symbol Tree → Functions 폴더. main이 보이면 더블클릭. 없으면 _start나 entry를 찾아서 main으로 들어가는 call을 따라갑니다.
  4. Decompiler 창 활용
    우측 Decompiler 패널에서 C 유사 코드로 확인합니다. 변수/함수 이름을 직접 편집(L키 = 레이블 변경, Y키 = 타입 변경)해서 분석 내용을 주석으로 남길 수 있습니다.
  5. 문자열 참조로 로직 찾기
    Search → For Strings → "password", "flag", "correct" 등을 검색. 찾은 문자열을 더블클릭하면 해당 위치로 이동합니다. 우클릭 → References → Show References to 클릭하면 그 문자열을 사용하는 함수를 찾을 수 있습니다.

📌 하드웨어 브레이크포인트(watch)란?

하드웨어 브레이크포인트(Hardware Breakpoint)는 CPU의 디버그 레지스터를 이용해 특정 메모리 주소의 읽기/쓰기를 감시합니다. 소프트웨어 브레이크포인트는 코드에 0xCC(INT3 명령어)를 삽입하지만, 하드웨어 브레이크포인트는 코드를 수정하지 않습니다. 안티디버깅 회피에 유용합니다. x86-64에서 최대 4개까지 설정 가능합니다.

버퍼(Buffer)란?

버퍼 (Buffer)

데이터를 임시로 저장하는 메모리 공간입니다. 예: char buf[64]는 64바이트짜리 버퍼입니다. 스택이나 힙에 할당됩니다. 버퍼 오버플로우(Buffer Overflow)는 버퍼 크기를 초과해서 데이터를 쓰는 것입니다. 초과된 데이터는 인접한 메모리(리턴 주소 등)를 덮어씁니다.

스택 기반 버퍼 오버플로우 (Stack BOF)

가장 고전적인 취약점입니다. 스택에 할당된 버퍼를 넘쳐서 리턴 주소를 덮어씁니다.

void vulnerable() {
    char buf[64];     // 스택에 64바이트 버퍼 할당
    gets(buf);        // ★ 취약! 입력 길이 제한 없음
    // scanf("%s", buf)도 마찬가지로 위험
}
높은 주소
rbp+8
리턴 주소 ← 덮어쓰기 목표!
★ 공격
rbp
saved rbp (8바이트)
rbp-64
buf[64] ← 여기서부터 gets() 입력 받음
버퍼
rsp
…

64바이트(buf) + 8바이트(saved rbp) = 72바이트를 A로 채운 뒤, 이어서 8바이트에 원하는 주소를 넣으면 ret 시 그 주소로 점프합니다.

# pwntools: Python 기반 CTF 익스플로잇 라이브러리
from pwn import *

p = process('./vulnerable')

# payload = 64 + 8 바이트 채우기 + 원하는 주소
payload = b'A' * 72           # buf(64) + saved_rbp(8)
payload += p64(0x401234)    # 원하는 함수 주소 (리틀엔디안 8바이트)

p.sendline(payload)
p.interactive()             # 셸 획득 후 상호작용
📌 셸코드(Shellcode)란?

셸코드(Shellcode)는 공격자가 실행하고 싶은 어셈블리 코드를 바이트로 만든 것입니다. 이름처럼 보통 /bin/sh(셸, 명령줄)를 실행하는 것이 목표입니다. NX(Non-Executable Stack) 보호가 없다면 버퍼에 셸코드를 넣고 버퍼 주소로 리턴하면 됩니다.

보호 기법들 — 현대 바이너리의 방어

ASLR
Address Space Layout Randomization
실행할 때마다 스택/힙/라이브러리 주소 무작위화. 하드코딩된 주소를 쓸 수 없게 합니다.
NX / DEP
Non-Executable Stack / Data Execution Prevention
스택/힙 메모리를 실행 불가로 표시. 셸코드 직접 실행을 막습니다.
Stack Canary
스택 카나리 (Stack Guard)
리턴 주소 앞에 랜덤 값(카나리)을 심어두고, ret 전에 변조 여부 확인. 변경 시 프로그램 즉시 종료.
PIE
Position Independent Executable
실행 파일 자체도 랜덤 주소에 로드. ASLR과 함께 코드 영역도 예측 불가.
RELRO
Relocation Read-Only
GOT 영역을 읽기 전용으로 잠그는 보호. Full RELRO면 GOT 덮어쓰기 공격 불가.
CFI
Control Flow Integrity
간접 호출/점프가 허용된 목적지만 가도록 컴파일러가 체크 코드 삽입. ROP 방어.

스택 카나리 동작 방식

rbp+8
리턴 주소
rbp
saved rbp
rbp-8
카나리 값 (랜덤, 프롤로그에서 삽입)
★ 감시
rbp-72
buf[64] ← 버퍼
vulnerable:
    ; 프롤로그
    push rbp
    mov  rbp, rsp
    sub  rsp, 72              ; buf(64) + 카나리(8)

    ; ★ 카나리 삽입 (프롤로그 마지막)
    mov  rax, fs:[0x28]       ; fs:0x28 = TLS에 저장된 랜덤 카나리 값
    mov  [rbp-8], rax         ; 스택에 카나리 저장
    xor  eax, eax

    ; ... 함수 본문 ...

    ; ★ 카나리 검사 (에필로그 시작)
    mov  rdx, [rbp-8]         ; 스택의 카나리 읽기
    xor  rdx, fs:[0x28]       ; 원래 값과 XOR
    jnz  __stack_chk_fail     ; 다르면 (변조됐으면) 프로그램 종료!
    leave
    ret

ROP (Return-Oriented Programming)

ROP (Return-Oriented Programming)

NX(Non-Executable Stack)로 셸코드를 직접 실행할 수 없을 때 쓰는 공격 기법입니다. 프로그램 안에 이미 존재하는 짧은 코드 조각들(가젯, Gadget — ret으로 끝나는 1~5개 명령어)을 스택에 쌓아서 연결합니다. 각 가젯이 실행되고 ret으로 다음 가젯으로 이어집니다. 마치 레고 블록처럼 가젯을 조합해서 원하는 동작을 만들어냅니다.

; 가젯 1: pop rdi; ret  (주소: 0x401234)
;   rdi에 값을 넣는 용도 → 함수 1번째 인수 설정에 사용
pop   rdi
ret

; 가젯 2: ret  (주소: 0x401200)  ← 스택 정렬용
ret

; 가젯 3: syscall  (주소: 0x401256)
syscall
ret

; ── ROP 체인 (스택에 이 순서로 배치) ──
; [리턴 주소 위치] 0x401234   → pop rdi; ret 실행
; [+8]             /bin/sh 주소 → rdi에 들어감
; [+16]            0x401200   → ret (정렬)
; [+24]            system() 주소 → system("/bin/sh") 실행!
📌 ROP 가젯 찾는 도구

ROPgadget과 ropper로 바이너리에서 가젯을 자동으로 찾습니다.

ROPgadget --binary ./prog --rop    # 가젯 목록
ROPgadget --binary ./prog --string "/bin/sh"  # 문자열 주소

CTF 리버싱 문제 접근법

CTF (Capture The Flag)

해킹 대회 형식입니다. 문제를 풀어 숨겨진 플래그(flag) 문자열을 찾아 제출하면 점수를 얻습니다. 리버싱, 취약점 분석, 암호화, 포렌식 등 다양한 분야가 있습니다. picoCTF, pwn.college, crackmes.one이 입문에 좋습니다.

  1. file, checksec으로 기본 파악 — 아키텍처, 보호 기법 확인. 어떤 보호가 있는지에 따라 공략 방향이 달라집니다.
  2. strings로 힌트 탐색 — "flag", "correct", "wrong", "password", "key" 등. 평문 정답이 들어있을 수도 있습니다.
  3. Ghidra 임포트 → main 찾기 — 자동 분석 후 문자열 참조로 핵심 함수 찾기. 함수/변수에 이름 붙이기.
  4. 비교 로직 찾기 — strcmp, cmp, test + 조건 점프 패턴. "정답"과 "오답" 메시지 참조 역추적.
  5. GDB로 동적 확인 — 비교 직전 브레이크포인트 설정. 실제 비교 대상 값을 레지스터에서 직접 읽기.
  6. z3 / angr 활용 — 복잡한 수백 번의 조건을 자동으로 만족하는 입력값 계산.
