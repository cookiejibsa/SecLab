Part 1 · Chapter 01

컴퓨터 기본 구조

어셈블리어는 CPU가 직접 이해하는 언어입니다. 그러므로 CPU가 어떻게 생겼는지, 메모리와 어떻게 대화하는지 먼저 알아야 합니다.

컴퓨터는 무엇으로 이루어져 있나?

컴퓨터는 크게 CPU(중앙처리장치), 메모리(RAM), 저장장치(SSD/HDD), 입출력장치로 이루어집니다.

CPU (Central Processing Unit, 중앙처리장치)

컴퓨터의 "뇌"입니다. 덧셈, 비교, 점프 같은 연산을 초당 수십억 번 실행합니다. 어셈블리 명령어는 결국 CPU에게 보내는 직접 지시입니다. "이 두 숫자를 더해라", "여기 주소의 값을 읽어라" 같은 명령들입니다.

RAM (Random Access Memory)

프로그램이 실행 중일 때 데이터를 임시로 저장하는 공간입니다. 책상 위의 메모지 같은 개념으로, 전원이 꺼지면 사라집니다. 어셈블리에서 mov로 읽고 쓰는 대부분의 공간이 이 RAM입니다. 주소(번지)로 접근하며, 모든 바이트는 고유한 주소를 가집니다.

저장장치 (SSD / HDD)

전원이 꺼져도 남아있는 영구 저장공간입니다. 파일들이 여기 있습니다. 프로그램을 실행하면 저장장치에서 RAM으로 복사되어 올라갑니다. SSD는 HDD보다 훨씬 빠르지만, 둘 다 RAM보다는 수십~수천 배 느립니다.

폰 노이만 구조 — 현대 컴퓨터의 설계 철학

폰 노이만 구조(Von Neumann Architecture)는 1945년에 만들어진 컴퓨터 설계 방식으로, 현재 여러분의 PC가 그대로 따르고 있습니다. 핵심 아이디어는 딱 하나입니다:

💡 폰 노이만의 핵심 아이디어

프로그램 코드(명령어)와 데이터를 같은 메모리에 저장한다. 덕분에 프로그램을 메모리에서 읽어와서 순서대로 실행할 수 있습니다. 어셈블리 코드도 결국 메모리의 특정 주소에 저장된 바이트들입니다.

비트(Bit)와 바이트(Byte)

비트 (Bit)

컴퓨터가 다룰 수 있는 가장 작은 정보 단위입니다. 0 또는 1, 두 가지 값만 가집니다. 왜 0과 1밖에 없냐면? 전기가 흐르면 1, 안 흐르면 0으로 물리적으로 표현하기 가장 쉽기 때문입니다.

바이트 (Byte)

8개의 비트를 묶은 단위입니다. 00000000부터 11111111까지, 즉 0부터 255까지의 값을 표현할 수 있습니다. 메모리의 주소는 바이트 단위로 매겨집니다. 1KB = 1024바이트, 1MB = 1024KB입니다.

📌 PTR이 뭔가요?

PTR은 Pointer(포인터)의 약자입니다. 여기서는 "이 주소에 있는 값을 ○○ 크기로 해석하라"는 뜻입니다. DWORD PTR [rbx]는 "rbx가 가리키는 주소에 있는 4바이트를 읽어라"라는 의미입니다.

수 체계와 진법

어셈블리 코드에는 숫자가 넘쳐납니다. 그런데 0x7fff, 0b1010 같은 이상한 표기가 나옵니다. 이게 뭔지 알아야 합니다.

우리가 쓰는 10진법(Decimal)

우리는 일상에서 10진법(Decimal)을 씁니다. 0~9까지 10개의 숫자를 쓰며, 9 다음에는 자리가 올라가 10이 됩니다. 예: 42, 255, 1000.

컴퓨터가 쓰는 2진법(Binary)

컴퓨터는 2진법(Binary)을 씁니다. 0과 1만 사용합니다. 1 다음에 바로 자리가 올라가서 10(이진수)이 됩니다.

어셈블리에서 자주 쓰는 16진법(Hexadecimal, Hex)

16진법(Hexadecimal)은 0~9, A~F까지 16개의 기호를 씁니다. A=10, B=11, C=12, D=13, E=14, F=15입니다. 왜 쓰냐고요? 2진수 4비트가 정확히 16진수 1자리에 대응되기 때문입니다. 8비트(1바이트)는 16진수 두 자리 FF로 표현됩니다. 훨씬 읽기 편합니다.

📌 표기법

C/어셈블리에서 0x를 앞에 붙이면 16진수입니다. 0xFF = 255, 0x41 = 65 = 문자 'A'. 메모리 주소는 대부분 0x7fffffffe5a0 같은 16진수로 표시됩니다.

음수 표현 — 2의 보수(Two's Complement)

컴퓨터는 음수를 어떻게 저장할까요? 2의 보수(Two's Complement) 방식을 사용합니다.

규칙은 간단합니다: 어떤 양수의 2의 보수를 구하려면, ① 비트를 모두 뒤집고 ② 1을 더합니다.

+5 = 0000 0101 (8비트 이진수)
 1단계: 비트 반전     → 1111 1010
 2단계: 1 더하기      → 1111 1011

따라서 -5 = 1111 1011 = 0xFB

왜 이 방식을 쓸까요? 덧셈 회로 하나로 덧셈과 뺄셈을 모두 처리할 수 있기 때문입니다. 5 + (-3)을 그냥 비트 덧셈으로 하면 정확히 2가 나옵니다. 컴퓨터 입장에서 매우 효율적입니다.

⚠️ 부호있는(Signed) vs 부호없는(Unsigned)

0xFF는 부호 없이 보면 255이고, 부호 있게 보면 -1입니다! 같은 비트 패턴이지만 해석이 달라집니다. 어셈블리에서 movsx(부호 확장)와 movzx(0 확장)의 차이가 여기서 생깁니다.

메모리 구조

프로그램이 실행될 때 메모리는 어떻게 나뉘어 있을까요? 이것을 모르면 스택 오버플로우, 버퍼 오버플로우 같은 개념을 이해할 수 없습니다.

메모리 주소란?

메모리 주소 (Memory Address)

RAM의 각 바이트는 고유한 번호(주소)를 가집니다. 아파트 호수처럼요. x86-64에서 주소는 64비트(8바이트)입니다. 보통 0x7fffffffe5a0 같은 16진수로 표시됩니다. 포인터(pointer)는 이 주소를 담는 변수입니다. C에서 int *p라고 쓸 때 p가 바로 주소를 담는 포인터입니다.

프로세스 메모리 레이아웃

텍스트(Text) 세그먼트 — 코드가 사는 곳

여러분이 작성한 C/어셈블리 코드가 컴파일되면, 기계어 바이트들이 이 구역에 저장됩니다. 읽기 전용입니다. Ghidra나 IDA로 바이너리를 열면 이 구역을 분석합니다. rip 레지스터(명령 포인터)가 현재 실행 중인 명령의 주소를 가리킵니다.

데이터(Data) / BSS 세그먼트 — 전역변수

전역변수가 저장되는 공간입니다. 초기값이 있는 것은 Data, 없는 것은 BSS에 들어갑니다.

// C 코드
int initialized = 42;   // Data 세그먼트 (초기값 있음)
int uninitialized;       // BSS 세그먼트 (0으로 초기화됨)
📌 BSS란?

BSS는 "Block Started by Symbol"의 약자입니다(역사적 이유로 이상한 이름이 됐습니다). 실행 파일에는 실제 0 데이터를 넣지 않고 "이 크기만큼 0으로 채워진 공간이 필요하다"는 정보만 넣습니다. 덕분에 실행 파일 용량이 줄어듭니다.

힙(Heap) — 동적으로 할당하는 메모리

C의 malloc()으로 요청하는 메모리가 여기서 옵니다. 힙은 낮은 주소에서 높은 주소 방향으로 자랍니다. 직접 해제(free())하지 않으면 메모리 누수(Memory Leak)가 발생합니다. 리버싱에서 힙 취약점(Heap Exploitation)이 중요한 이유입니다.

스택(Stack) — 함수가 쓰는 임시 공간

가장 중요합니다. 함수를 호출할 때마다 스택에 새 공간이 생기고, 함수가 끝나면 사라집니다. 높은 주소에서 낮은 주소 방향으로 자랍니다. 즉, push하면 rsp(스택 포인터)가 감소합니다. 이 역방향 성장이 버퍼 오버플로우 공격의 핵심 요소입니다.

✅ 핵심 암기

스택은 위에서 아래로 자랍니다 (주소 감소). 힙은 아래에서 위로 자랍니다 (주소 증가). push rax는 rsp를 8 감소시키고 rax 값을 씁니다.

엔디안(Endianness) — 바이트 저장 순서

리틀 엔디안 (Little Endian)

x86/x86-64가 사용하는 방식입니다. 숫자를 메모리에 저장할 때 작은(낮은) 바이트가 낮은 주소에 먼저 옵니다. 예를 들어 0x12345678이라는 4바이트 값은 메모리에 78 56 34 12 순서로 저장됩니다. "리틀"은 "작은 쪽(낮은 바이트)이 먼저"라는 의미입니다.

값: 0x12345678 (32비트 정수)

주소 0x100: 0x78         주소 0x100: 0x12
주소 0x101: 0x56         주소 0x101: 0x34
주소 0x102: 0x34         주소 0x102: 0x56
주소 0x103: 0x12         주소 0x103: 0x78
⚠️ 리버싱에서 중요

Ghidra나 xxd(헥스 덤프 도구)로 바이너리를 볼 때 바이트가 거꾸로 보일 수 있습니다. x86은 리틀 엔디안이므로 4바이트 이상의 값을 읽을 때 항상 뒤집어서 해석하세요.

Part 1 · Chapter 04

CPU 작동 원리

어셈블리 명령어 하나가 CPU에서 어떻게 처리되는지 이해하면, 왜 레지스터가 필요한지, 왜 캐시가 중요한지 자연스럽게 이해됩니다.

명령어 사이클 — CPU가 하는 일의 반복

CPU는 매우 단순한 일을 엄청 빠르게 반복합니다. 이 반복을 명령어 사이클(Instruction Cycle)이라고 합니다:

  1. 패치(Fetch) — 명령어 가져오기
    rip(명령 포인터 레지스터)가 가리키는 메모리 주소에서 다음 명령어(기계어 바이트)를 읽어옵니다. 예: 주소 0x401234에서 48 89 c7 같은 바이트를 읽습니다.
  2. 디코드(Decode) — 명령어 해석
    읽어온 바이트가 무슨 명령인지 해석합니다. 48 89 c7이 mov rdi, rax라는 명령임을 알아냅니다. 명령어마다 길이가 다릅니다(1~15바이트). 이 해석 과정을 디코딩이라고 합니다.
  3. 실행(Execute) — 명령 수행
    ALU(산술논리장치)가 실제 연산을 수행합니다. add rax, rbx라면 두 레지스터 값을 더하는 회로가 작동합니다.
  4. 라이트백(Writeback) — 결과 저장
    연산 결과를 레지스터나 메모리에 씁니다. 플래그 레지스터(rflags)도 이때 갱신됩니다.

캐시(Cache) — CPU와 RAM 사이의 고속 메모리

캐시 (Cache)

RAM은 CPU에 비해 매우 느립니다. 그래서 CPU 내부에 더 빠른 소용량 메모리(캐시)를 두고, 자주 쓰는 데이터를 미리 복사해둡니다. L1(가장 빠름, 작음) → L2 → L3 순으로 계층이 있습니다. 어셈블리 최적화에서 "캐시 히트/미스"가 중요한 성능 요소입니다.

메모리 유형속도용량위치
레지스터1 사이클수십 개CPU 내부
L1 캐시~4 사이클32~64KBCPU 내부
L2 캐시~12 사이클256KB~1MBCPU 내부
L3 캐시~40 사이클8~32MBCPU(공유)
RAM~200 사이클8~64GB별도 칩
SSD수만 사이클수백GB~TB별도 장치
📌 사이클이란?

사이클(Cycle)은 CPU 클럭 한 번의 단위입니다. 3GHz CPU는 1초에 30억 번 클럭이 뜁니다. 1 사이클 = 약 0.33나노초(0.00000000033초)입니다. 레지스터는 1사이클이면 끝나지만, RAM은 200사이클이나 기다려야 합니다. 그래서 레지스터 접근이 압도적으로 빠릅니다.

Part 1 · Chapter 05

운영체제 기초

어셈블리로 시스템 콜을 하거나, 리버싱으로 프로그램을 분석할 때 운영체제가 하는 역할을 알아야 합니다.

커널(Kernel)과 유저 공간(User Space)

커널 (Kernel)

운영체제의 핵심입니다. 하드웨어(메모리, 파일시스템, 네트워크)를 직접 관리하며, 특권 명령어를 실행할 수 있습니다. "링 0(Ring 0)"이라는 최고 권한 모드에서 실행됩니다.

유저 공간 (User Space)

우리가 만드는 프로그램이 실행되는 공간입니다. 직접 하드웨어를 건드릴 수 없고, 커널에 요청(시스템 콜)을 해야 합니다. "링 3(Ring 3)"에서 실행됩니다. 이 구분이 보안의 핵심입니다.

시스템 콜(System Call)

유저 프로그램이 커널에 서비스를 요청하는 방법입니다. C의 printf()는 내부적으로 write 시스템 콜을 씁니다. 어셈블리에서는 syscall 명령어로 직접 호출합니다. 자세한 사용법은 Chapter 15에서 다룹니다.

📌 왜 시스템 콜이 필요한가?

만약 모든 프로그램이 하드웨어를 마음대로 건드릴 수 있다면? 악성 프로그램이 다른 프로그램의 메모리를 훔쳐보거나, 디스크를 마음대로 지울 수 있습니다. 커널이 문지기 역할을 해서 "파일을 읽고 싶다면 나(커널)한테 요청해라"라고 통제합니다.

프로세스와 쓰레드

프로세스 (Process)

실행 중인 프로그램의 인스턴스입니다. 자체 메모리 공간(주소 공간)을 가집니다. 다른 프로세스의 메모리를 함부로 읽을 수 없습니다 (OS가 가상 메모리로 분리).

쓰레드 (Thread)

프로세스 안에서 동시에 실행되는 여러 실행 흐름 중 하나입니다. 같은 프로세스 내 쓰레드들은 메모리를 공유합니다. 예: 웹 브라우저에서 동영상 재생 쓰레드, 사용자 입력 처리 쓰레드가 동시에 돌아갑니다. 어셈블리에서는 각 쓰레드마다 고유한 스택이 있습니다.

가상 메모리 (Virtual Memory)

각 프로세스는 메모리 전체를 혼자 쓰는 것처럼 보이는 가상 주소 공간을 받습니다. 실제 RAM 주소와의 변환은 MMU(메모리 관리 장치)가 담당합니다. 그래서 다른 프로그램과 주소가 충돌하지 않습니다.

✅ 리버싱에서 중요한 이유

리버싱 도구(gdb, x64dbg)로 프로세스를 분석할 때 보이는 주소들은 가상 주소입니다. ASLR(주소 공간 배치 난수화)이 켜져 있으면 실행할 때마다 주소가 달라집니다. 이것도 보안 기법 중 하나입니다.

레지스터 완전 해부

레지스터는 CPU 안에 있는 초고속 저장 공간입니다. 어셈블리의 모든 연산은 레지스터를 중심으로 돌아갑니다.

레지스터란?

레지스터 (Register)

CPU 안에 있는 극소량의 저장 공간입니다. RAM보다 수백 배 빠릅니다. x86-64에는 16개의 범용 레지스터가 있으며, 각각 64비트(8바이트)입니다. C 변수를 메모리에 저장하는 것과 달리, 어셈블리에서는 가능한 한 레지스터를 이용해 연산합니다.

레지스터 크기별 이름 — 같은 물리 레지스터, 다른 이름

중요합니다! x86-64의 레지스터는 크기별로 다른 이름을 가집니다. rax의 하위 32비트가 eax, 하위 16비트가 ax, 하위 8비트가 al입니다. 이들은 같은 물리 공간의 일부입니다:

📌 이름의 의미

이름에는 역사가 있습니다. RAX = Register AX (64비트 확장), EAX = Extended AX (32비트), AX = 16비트 원래 이름, AH = AX High (상위 8비트), AL = AX Low (하위 8비트). e(Extended/확장), r(Register/64비트 시대)이 붙은 것입니다.

⚠️ 32비트 쓰면 상위가 0이 됩니다

mov eax, 1을 실행하면 rax의 상위 32비트는 자동으로 0이 됩니다. 이건 x86-64의 중요한 특성입니다. 반면 mov ax, 1은 상위 48비트를 건드리지 않습니다.

기본 명령어

어셈블리 명령어는 수백 개가 있지만, 실제 코드의 80%는 여기 나오는 명령어들로 이루어집니다. 각 명령어를 C와 1:1로 대응시켜 이해합시다.

어셈블리 명령어 기본 형식

명령어  목적지, 소스
; 예:
mov  rax, rbx    ; rax = rbx  (rbx를 rax로 복사)
add  rax, 5      ; rax = rax + 5
sub  rdi, rsi    ; rdi = rdi - rsi
📌 Intel 문법 vs AT&T 문법

어셈블리에는 두 가지 문법이 있습니다. Intel 문법: mov rax, rbx (목적지 먼저), AT&T 문법: movq %rbx, %rax (소스 먼저, % 붙음). 이 교재는 Intel 문법을 씁니다. Ghidra도 기본적으로 Intel 문법입니다. GDB는 기본이 AT&T이지만 set disassembly-flavor intel로 바꿀 수 있습니다.

MOV — 데이터 이동

mov는 어셈블리에서 가장 많이 쓰이는 명령어입니다. "이동"이라는 이름이지만 실제로는 복사입니다. 원본은 그대로 남습니다.

C 코드
int a = 42;
int b = a;
long c = 0x1234567890;
어셈블리 (Intel)
mov  eax, 42
mov  ebx, eax
mov  rcx, 0x1234567890
명령어의미예시
mov dst, srcsrc 값을 dst로 복사mov rax, rbx
movsx dst, src복사 + 부호 확장 (음수 보존)movsx rax, eax
movzx dst, src복사 + 0으로 확장movzx rax, al
lea dst, [src]주소 계산 결과를 dst에 저장lea rax, [rbx+8]
xchg dst, src두 값 교환 (swap)xchg rax, rbx

LEA — 주소 계산의 만능 도구

LEA (Load Effective Address)는 겉보기엔 주소를 계산하는 명령어이지만, 컴파일러가 곱셈/덧셈을 할 때도 씁니다!

C 코드
// 포인터 주소 계산
int *p = arr + i;

// 곱셈 (최적화 시)
int x = a * 3;
// = a + a*2
어셈블리
; rbx = arr 주소, rax = i
lea  rcx, [rbx + rax*4]

; rax = a
lea  rax, [rax + rax*2]
; rax = a + a*2 = a*3

산술 명령어

C 코드
a = a + b;
a = a - b;
a = a * b;
a = a / b;  // 몫
c = a % b;  // 나머지
a++;
a--;
a = -a;
어셈블리
add  rax, rbx
sub  rax, rbx

⚠️ idiv 주의사항

idiv rbx는 rdx:rax(128비트 값으로 취급)를 rbx로 나눕니다. 나누기 전에 반드시 rdx를 설정해야 합니다. 양수를 나눈다면 xor rdx, rdx(부호없음) 또는 cqo(부호있음 확장)를 먼저 하세요.

비트 연산 명령어

비트 연산은 암호화, 플래그 체크, 최적화에서 자주 나옵니다.

명령어C 연산자설명예시 (rax=0b1010, rbx=0b1100)
and&둘 다 1이면 1, 나머지 0and rax, rbx → 0b1000
or|하나라도 1이면 1or rax, rbx → 0b1110

XOR 자기 자신 — 0 초기화 트릭

xor rax, rax는 rax를 0으로 만드는 관용구입니다. mov rax, 0보다 기계어가 짧아서 컴파일러가 자주 씁니다. 리버싱에서 자주 보이므로 반드시 외우세요.

xor  eax, eax   ; rax = 0 (가장 짧은 0 초기화)
xor  rcx, rcx   ; rcx = 0

비교 명령어 — CMP와 TEST

비교는 실제로 빼기/AND를 수행하지만, 결과를 저장하지 않고 플래그만 바꿉니다. 다음 조건 점프 명령어가 이 플래그를 읽습니다.

C 코드
if (a == b) { ... }
if (a != 0) { ... }
if (flags & 0x1) { ... }
어셈블리
cmp  rax, rbx  ; a-b 계산, 플래그만 변경
je   .equal  ; ZF=1이면 점프
test rax, rax  ; rax & rax, ZF 설정
jnz  .nonzero
test rax, 0x1  ; 비트 체크
jnz  .bit_set

메모리 접근 — 포인터의 어셈블리 버전

C에서 포인터를 어렵게 느꼈다면, 어셈블리에서 메모리 접근 방식을 보면 오히려 명확하게 이해됩니다. 대괄호 [ ]가 C의 역참조(dereference) *와 같습니다.

대괄호 [ ] = C의 * (역참조)

C 코드
int x = 10;
int *p = &x;

// 포인터로 읽기
int val = *p;

// 포인터로 쓰기
*p = 20;
어셈블리
; rbx = p (포인터, 주소를 가짐)

; 포인터로 읽기 (역참조)
mov  rax, [rbx]
; = rax = *p

; 포인터로 쓰기
mov  QWORD PTR [rbx], 20

메모리 주소 계산 공식

Intel x86-64의 메모리 주소는 다음 공식으로 계산됩니다:

[base + index * scale + displacement]
 베이스    인덱스   배율(1,2,4,8)   오프셋(변위)
📌 오프셋(Offset)이란?

오프셋(Offset)은 "기준 위치에서 얼마나 떨어져 있는가"입니다. 예를 들어 구조체의 두 번째 멤버는 시작 주소에서 4바이트 떨어진 곳(오프셋 4)에 있습니다. 디스플레이스먼트(Displacement)는 어셈블리에서 같은 개념을 부르는 이름입니다.

표현식계산되는 주소C에 대응
[rbx]rbx*p
[rbx + 8]rbx + 8*(p + 2) (int 기준)
[rbx + rax*4]rbx + rax*4arr[i] (int 배열)
[rbp - 8]rbp - 8로컬 변수
[rbx + rax*8 + 16]rbx + rax*8 + 16구조체 배열 접근

크기 지정자 — BYTE PTR, WORD PTR, DWORD PTR, QWORD PTR

*p1 = 0x41;  // 1바이트
*p2 = 0x1234; // 2바이트
*p3 = 0x5678; // 4바이트
*p4 = 0x9ABC; // 8바이트

플래그 레지스터와 조건 점프

어셈블리에서 if문과 반복문은 플래그와 조건 점프 명령어로 만들어집니다. 이것이 어셈블리의 제어 흐름(Control Flow)입니다.

RFLAGS — 상태 플래그 레지스터

RFLAGS는 연산 결과의 "상태"를 기억하는 특별한 레지스터입니다. 각 비트가 하나의 플래그를 나타냅니다. cmp나 add 같은 연산 후 자동으로 갱신됩니다.

ZF
Zero Flag
연산 결과가 0이면 1. cmp a, b에서 a==b이면 두 값의 차가 0이므로 ZF=1.
CF
Carry Flag
덧셈에서 올림수(carry) 발생 시, 또는 뺄셈에서 빌림(borrow) 발생 시 1. 부호없는 비교에 사용.
SF
Sign Flag
결과의 최상위 비트(부호 비트)와 같습니다. 음수면 SF=1.
OF
Overflow Flag
부호있는 연산에서 오버플로우(표현 범위 초과) 발생 시 1.
PF
Parity Flag
결과의 하위 8비트에서 1인 비트 수가 짝수면 1. 현대에는 잘 안 씁니다.
DF
Direction Flag
문자열 명령어 방향. 0=증가, 1=감소. cld/std로 설정.

C의 if-else를 어셈블리로

C 코드
if (x > 0) {
    result = x * 2;
} else {

어셈블리 (rdi=x, rax=result)

📌 레이블(Label)이란?

레이블(Label)은 코드의 특정 위치에 붙이는 이름표입니다. .else:, .end: 같이 콜론(:)으로 끝납니다. 점프 명령어가 이 이름을 목적지로 사용합니다. 실제 기계어에서는 주소로 변환됩니다.

    sum += i;
}
어셈블리 (rax=sum, rcx=i)

스택(Stack) 동작 상세

스택은 함수 호출, 지역 변수, 리턴 주소를 관리하는 핵심 구조입니다. 스택을 완전히 이해하면 버퍼 오버플로우 공격도 이해됩니다.

스택이란? — LIFO 자료구조

LIFO (Last In, First Out)

마지막에 넣은 것이 먼저 나옵니다. 책상에 쌓은 책처럼요. push로 맨 위에 놓고, pop으로 맨 위에서 꺼냅니다. rsp(스택 포인터)는 항상 현재 스택의 "맨 위"(가장 낮은 주소)를 가리킵니다. LIFO는 자료구조의 일종입니다.

PUSH와 POP의 정확한 동작

PUSH rax 동작
// 등가 C 코드
rsp -= 8;        // 스택 포인터 감소
*(long*)rsp = rax; // 값 저장
POP rax 동작
// 등가 C 코드
rax = *(long*)rsp; // 값 읽기
rsp += 8;         // 스택 포인터 증가

스택은 높은 주소에서 낮은 주소로 자랍니다. push할수록 rsp는 작아집니다.

CALL과 RET — 함수 호출의 본질

CALL 명령어

call func는 두 가지 일을 합니다: ① 현재 rip(다음 명령어 주소)를 스택에 push, ② func 주소로 점프. 즉 push rip; jmp func와 같습니다. 스택에 저장된 이 주소를 리턴 주소(Return Address)라고 합니다. 함수가 끝난 후 돌아올 위치를 기억하는 것입니다.

RET 명령어

ret는 스택 최상단 값을 rip에 넣고 rsp를 8 증가시킵니다. 즉 pop rip와 같습니다. call이 저장해둔 리턴 주소로 돌아갑니다.

🔴 보안: 버퍼 오버플로우의 원리

스택에는 "로컬 변수"와 "리턴 주소"가 인접해 있습니다. 로컬 버퍼(예: 64바이트 배열)에 경계 검사 없이 데이터를 쓰면, 버퍼를 넘쳐서 리턴 주소를 덮어쓸 수 있습니다. ret 시 공격자가 원하는 주소로 점프됩니다. 이것이 고전적인 스택 기반 버퍼 오버플로우(Stack-based Buffer Overflow)입니다. 자세한 내용은 Chapter 20에서 다룹니다.

함수 호출 규약 (Calling Convention)

여러 함수가 통신하려면 약속이 필요합니다. "인수를 어디에 넣어서 넘길까?", "리턴값은 어디에?", "어떤 레지스터를 보존해야 하나?" 이 약속이 호출 규약입니다.

📌 인수(Argument)와 매개변수(Parameter)의 차이

매개변수(Parameter): 함수를 정의할 때 쓰는 변수 이름입니다. int add(int a, int b)에서 a, b가 매개변수.
인수(Argument): 함수를 호출할 때 실제로 전달하는 값입니다. add(3, 4)에서 3, 4가 인수.

System V AMD64 ABI — Linux/macOS의 규약

Linux와 macOS는 System V AMD64 ABI(Application Binary Interface)를 따릅니다. Windows는 다릅니다(아래 참고). 리버싱에서 리눅스 바이너리를 많이 다루므로 이것을 먼저 외웁니다.

리턴값

정수/포인터 리턴값은 rax에, 128비트 리턴은 rax:rdx에 넣습니다. 부동소수점 리턴은 xmm0에 넣습니다.

Caller-saved vs Callee-saved 레지스터

Caller-saved (호출자가 저장)

함수를 호출하기 전에 내가(caller가) 저장해야 하는 레지스터입니다. 피호출 함수가 마음대로 수정할 수 있습니다. rax, rcx, rdx, rsi, rdi, r8, r9, r10, r11. Caller는 '부르는 쪽', Callee는 '불리는 쪽'입니다.

Callee-saved (피호출자가 저장)

호출된 함수가 사용하기 전에 저장하고, 리턴 전에 복원해야 하는 레지스터입니다. 호출자는 이 레지스터들이 함수 호출 전후로 변하지 않는다고 믿을 수 있습니다. rbx, rbp, r12, r13, r14, r15

C 코드
int add(int a, int b) {
    return a + b;
}

int main() {
    int r = add(3, 4);
}
어셈블리
add:                   ; int add(int a, int b)
    lea  eax, [edi+esi]  ; a+b → eax (리턴값)
    ret

main:
    mov  edi, 3          ; 1번째 인수 a=3
    mov  esi, 4          ; 2번째 인수 b=4
    call add             ; 함수 호출
    ; rax = 7 (리턴값)

함수 프롤로그(Prologue)와 에필로그(Epilogue)

모든 함수는 시작과 끝에 정해진 패턴을 따릅니다. 이것을 프롤로그(시작 의식)와 에필로그(끝 의식)라고 합니다. 리버싱에서 함수 경계를 찾을 때 이 패턴을 봅니다.

C 함수
long example(long x) {
    long a = x * 2;
    long b = a + 1;
    return b;
}
어셈블리 (rdi=x)
example:
    ; ── 프롤로그 ──
    push rbp         ; 이전 rbp 저장
    mov  rbp, rsp   ; rbp = 현재 스택 위치
    sub  rsp, 16    ; 로컬변수 공간 16바이트 확보
    ; ── 함수 본문 ──
    lea  rax, [rdi*2]
    mov  [rbp-8],  rax  ; a = x*2
    add  rax, 1
    mov  [rbp-16], rax  ; b = a+1
    mov  rax, [rbp-16]  ; 리턴값 준비
    ; ── 에필로그 ──
    leave            ; mov rsp,rbp; pop rbp
    ret              ; 리턴 주소로 점프
📌 leave 명령어

leave는 mov rsp, rbp + pop rbp를 합친 단축 명령어입니다. 에필로그에서 스택을 정리할 때 씁니다. enter 명령어도 있지만 거의 사용 안 합니다.

Frame Pointer Omission (FPO) — 최적화 시

컴파일러가 최적화(-O2)를 하면 rbp를 프레임 포인터로 쓰지 않습니다. 대신 rsp 기준으로 지역변수를 접근합니다. 프롤로그에서 push rbp; mov rbp, rsp가 없어집니다. 리버싱에서 프레임이 없는 함수를 만날 수 있으니 알아두세요.

배열과 구조체 접근

C의 배열과 구조체가 어셈블리에서 어떻게 표현되는지 알면, 리버싱 시 데이터 구조를 복원하는 데 큰 도움이 됩니다.

배열 접근

배열은 연속된 메모리 블록입니다. arr[i]는 arr 시작 주소 + i * sizeof(element)입니다.

C 코드
int arr[5] = {10,20,30,40,50};
int x = arr[2];    // x = 30
arr[i] = 99;       // i번째 설정
long la[3];
la[i] = 100;       // 8바이트 원소
어셈블리 (rbx=arr, rcx=i)
; int arr[]: 원소 4바이트
mov  eax, [rbx + 2*4]   ; arr[2]
mov  [rbx + rcx*4], 99  ; arr[i]=99

; long arr[]: 원소 8바이트
mov  QWORD PTR [rbx + rcx*8], 100

구조체 접근

구조체는 멤버들이 순서대로 메모리에 배치됩니다. 각 멤버는 베이스 주소에 오프셋(offset)으로 접근합니다.

C 코드
struct Point {
    int   x;    // offset 0, 4바이트
    int   y;    // offset 4, 4바이트
    long  z;    // offset 8, 8바이트
};             // 합계: 16바이트

struct Point p;
p.x = 10;
p.y = 20;
p.z = 30;
int a = p.x + p.y;
어셈블리 (rbx=&p)
; p.x = 10  (offset 0)
mov  DWORD PTR [rbx], 10
; p.y = 20  (offset 4)
mov  DWORD PTR [rbx+4], 20
; p.z = 30  (offset 8)
mov  QWORD PTR [rbx+8], 30

; a = p.x + p.y
mov  eax, [rbx]      ; p.x
add  eax, [rbx+4]   ; + p.y

구조체 패딩(Padding)

컴파일러는 성능을 위해 멤버들 사이에 빈 바이트(패딩)를 넣습니다. 정렬(Alignment) 요구사항 때문입니다. 4바이트 int는 4의 배수 주소에, 8바이트 long은 8의 배수 주소에 있어야 합니다.

struct BadLayout {
    char  a;   // offset 0, 1바이트
               // 패딩 3바이트 (int 정렬 맞추기용)
    int   b;   // offset 4, 4바이트
    char  c;   // offset 8, 1바이트
               // 패딩 7바이트 (long 정렬 맞추기용)
    long  d;   // offset 16, 8바이트
};             // sizeof = 24 (char 3개뿐인데!)
✅ 리버싱 팁

구조체를 리버싱으로 복원할 때 오프셋 패턴을 보면 패딩을 고려해야 합니다. Ghidra의 Data Type Manager에서 구조체를 정의할 때 패딩도 넣어줘야 정확한 분석이 됩니다.

문자열 조작 명령어

x86에는 문자열(바이트 배열)을 다루는 전용 명령어가 있습니다. memcpy, memset, strlen 같은 함수들이 내부적으로 이 명령어를 쓸 수 있습니다.

📌 memcpy, memset, strlen이란?

memcpy(dst, src, n): src에서 dst로 n바이트를 복사합니다 (메모리 복사).
memset(ptr, val, n): ptr부터 n바이트를 모두 val로 채웁니다 (메모리 채우기). 보통 0으로 초기화할 때 사용.
strlen(str): 문자열 str의 길이를 반환합니다. 널 문자(\0)가 나올 때까지 셉니다.

REP — 반복 실행 접두사

REP 접두사

접두사는 명령어 앞에 붙여서 동작을 수정합니다. rep는 명령어 앞에 붙여서 rcx번 반복합니다. rcx가 0이 될 때까지 명령어를 실행하고, 매 실행마다 rcx--합니다. repe(ZF=1인 동안)와 repne(ZF=0인 동안)도 있습니다.

주요 문자열 명령어

명령어동작C 유사 함수
movsb/w/d/q[rsi] → [rdi], rsi++, rdi++ (또는 DF=1이면 --)memcpy 한 원소
rep movsbrcx바이트 복사 (rsi→rdi)memcpy(rdi, rsi, rcx)
stosb/w/d/qal/ax/eax/rax → [rdi], rdi++memset 한 원소
rep stosbrcx바이트를 al로 채움memset(rdi, al, rcx)
scasb/w/d/qal − [rdi], ZF 설정, rdi++검색/비교
repne scasbZF=0인 동안(al≠[rdi]) 반복strlen 구현 가능
cmpsb/w/d/q[rsi] − [rdi], 플래그 설정memcmp 한 원소
📌 접미사 b/w/d/q의 의미

b = byte (1바이트), w = word (2바이트), d = dword (4바이트), q = qword (8바이트). movsb는 1바이트씩, movsq는 8바이트씩 복사합니다.

C: strlen 구현
size_t strlen(const char *s) {
    const char *p = s;
    while (*p != '\0') p++;
    return p - s;
}
어셈블리 (rdi=s)
strlen:
    mov  rcx, -1     ; 최대 길이 (-1 = 매우 큰 수)
    xor  al, al      ; al = 0 (널문자)
    cld              ; DF=0 (증가방향)
    repne scasb      ; al≠[rdi]인 동안 반복
    ; rcx = -(len+2)
    not  rcx         ; 비트반전
    dec  rcx         ; -1 (널문자 빼기)
    mov  rax, rcx    ; 리턴값
    ret

Hello World — 순수 어셈블리

; hello.asm — NASM 문법, Linux x86-64
; NASM(Netwide Assembler): 대표적인 x86 어셈블러 중 하나입니다.
section .data
    msg db "Hello, World!", 0x0a  ; 문자열 + 줄바꿈(0x0a = '\n')
    len equ $ - msg               ; $ = 현재 위치, len = 문자열 길이

section .text
    global _start    ; 프로그램 시작점 선언 (_start가 main 역할)

_start:
    ; write(1, msg, len)
    mov  rax, 1       ; syscall: write (번호 1)
    mov  rdi, 1       ; fd = 1 (stdout, 표준출력)
    mov  rsi, msg     ; 버퍼 주소
    mov  rdx, len     ; 길이
    syscall

    ; exit(0)
    mov  rax, 60      ; syscall: exit (번호 60)
    xor  rdi, rdi    ; 종료코드 0
    syscall
📌 fd(File Descriptor)란?

fd(File Descriptor, 파일 서술자)는 열린 파일을 가리키는 숫자입니다. 리눅스에서는 파일, 소켓, 파이프 모두 fd로 다룹니다. 기본적으로 0=표준입력(stdin), 1=표준출력(stdout), 2=표준에러(stderr)입니다. write(1, ...)은 "화면(stdout)에 써라"는 뜻입니다.

링킹(Linking)은 컴파일된 여러 오브젝트 파일(.o)과 라이브러리를 합쳐 실행 가능한 파일을 만드는 과정입니다. ld가 링커입니다. 리버싱에서 ldd ./binary 명령으로 바이너리가 어떤 라이브러리에 의존하는지 볼 수 있습니다.

컴파일러 최적화 패턴

실제 바이너리를 리버싱하면 사람이 직접 짠 어셈블리와 달리 이상하게 생긴 코드들이 나옵니다. 이것들은 컴파일러 최적화의 결과입니다. 패턴을 알면 원래 C 코드를 추론할 수 있습니다.

나눗셈 → 곱셈+시프트 변환

나눗셈(idiv)은 매우 느린 명령어입니다. 컴파일러는 상수로 나누는 경우 곱셈과 시프트로 대체합니다.

C 코드 (원본 의도)
int x = n / 7;
컴파일러가 생성한 코드
; n / 7 최적화
mov   eax, edi         ; n
mov   ecx, 0x92492493  ; 마법의 상수(Magic Number)
imul  ecx              ; edx:eax = n * 상수
add   edx, edi
sar   edx, 2            ; 오른쪽 시프트
sar   edi, 31           ; 부호 처리
sub   edx, edi
; edx = n / 7
✅ 리버싱 팁

이런 "마법의 상수" 패턴을 보면 나눗셈입니다. 온라인 도구 "MagicDivider Calculator"나 Ghidra의 디컴파일러가 자동으로 원래 나눗셈으로 복원해줍니다.
